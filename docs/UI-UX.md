# ZeRo OS Web UI 设计规范

> **文档关系说明**：本文档定义 Web UI 的视觉风格、交互设计和页面规范。技术选型（框架、组件库）和命名规范以 [TechStack] 文档为准。本文档中的代码示例（CSS、API 路由）仅作设计说明用途，实际实现以 [TechStack - 命名规范] 和 [TechStack - Web UI 技术栈] 为准。

## 定位

Web UI 是 ZeRo OS 的控制面板 + 观测台，同时作为轻量 Channel。

- **主要功能**：观测系统状态、查看历史、管理配置
- **次要功能**：Web Channel（侧边抽屉形式的轻量对话）
- 飞书 / Telegram 是主力交互入口，Web UI 是辅助

安静的未来感——不是赛博朋克的霓虹喧嚣，而是深空望远镜控制台的气质。Linear 的克制 + Vercel 的质感 + 天文台控制室的氛围。

### 动画原则：有意义的运动

**页面进入 — 渐次浮现**

```css
.card {
  animation: fadeUp 0.4s ease-out forwards;
}
.card:nth-child(n) { animation-delay: calc(n * 60ms); }

@keyframes fadeUp {
  from { opacity: 0; transform: translateY(8px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

**实时数据 — Token 数字滚动**

数字更新时每位独立滚动，像机械翻牌器，让"系统正在消耗资源"可感知。

**活跃 Session — 呼吸脉搏**

```css
.status-active {
  animation: pulse 2.5s ease-in-out infinite;
}
@keyframes pulse {
  0%, 100% { box-shadow: 0 0 6px rgba(52, 211, 153, 0.4); }
  50%      { box-shadow: 0 0 12px rgba(52, 211, 153, 0.6); }
}
```

**Chat Drawer — 侧滑 + 磨砂遮罩**

```css
.drawer-panel {
  animation: slideIn 0.3s cubic-bezier(0.16, 1, 0.3, 1) forwards;
}
```

Apple 风格弹性缓出曲线。

**不做动画的地方**

- 页面切换（直接渲染，不要过渡延迟）
- 表格排序/筛选（数据变更要即时）
- Hover 状态（`0.15s ease` 足够）

## Dashboard 页面

### 信息层级（从上到下，从紧急到背景）

1. 系统还活着吗？（0.5 秒确认）
2. 有没有需要处理的？（1 秒扫到）
3. 现在在干什么？（活跃 Session）
4. 刚才发生了什么？（事件流）
5. 花了多少钱？（费用概览）

### A. 系统状态栏

页面顶部窄带（40px），正常时几乎隐形，异常时醒目：

```

熔断:  ◉ Fused      等待人工介入                    ← 红色，脉搏加快
```

卡片边框用 `amber-400` 微弱光晕，第一眼就能注意到。

```
│  ┌───────────────────────────────────────────────────────────┐ │
```

### E. 事件流

```
┌─ Recent Activity ──────────────────────────────────────────────┐
│  10:05   sess_001  ▸ bash    git commit -m "refactor adapter"  │
│  10:04   sess_001  ▸ edit    src/provider.ts L42-58            │
│  09:00   sess_003  ▸ sched   weekly_report 触发                │
│  08:30   system    ▸ alert   claude-opus API 超时，降级到 sonnet│
│                                          查看全部 →             │
└────────────────────────────────────────────────────────────────┘
```

- 单行时间线，信息密度高
- 新事件从顶部滑入（`translateY(-12px) → 0`）
- 系统级事件用状态色高亮
- 实时更新（WebSocket）

### 动态布局

Dashboard 根据系统状态自适应：

- 一切正常时：B 卡片不出现，页面简洁
- 有待办时：B 和 C 并排
- 系统异常时：B 提升到全宽，状态栏变色

工具调用块结构：

```
┌─ [工具图标] [工具名] ─── [状态] ──── [耗时] ───┐
```

- 新消息从底部出现，带淡入动画
- Agent 回复有 streaming 效果
- 工具调用进行中显示 `⟳` + 脉搏边框
- 自动滚动到底部（用户上翻时停止，显示"跳到最新 ↓"）

#### 排队消息的展示

当用户在 Agent 执行任务期间发来消息（详见 [Architecture - 消息排队]），时间线上需要特殊标识：

**排队确认**：用户发送消息后，如果 Agent 正忙，消息气泡正常显示，下方追加一行系统提示：

```
User 消息:
┌──────────────────────────────────────────────────────┐
│  现在处理得怎么样了                                    │
│  ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄  │
│  ⏳ 消息已排队，当前工具执行完毕后处理                  │
└──────────────────────────────────────────────────────┘
```

提示行用 `slate-500` 文字 + `⏳` 图标，不抢视觉焦点。当排队消息被 Agent 实际处理后，提示行自动消失。

**Agent 对排队消息的回应**：Agent 回应排队消息时，其回复在时间线上和正常回复视觉一致，但在回复气泡上方插入一条小标签：

```
   ┌─ 回应排队消息 ─────────────────────────────────────┐
   │  ↳ 对 "现在处理得怎么样了" 的回应                    │
   └───────────────────────────────────────────────────┘
   Agent 消息:
   目前已经完成了 git diff 分析，发现需要改 3 个文件...
```

标签用 `slate-600` 背景 + 小字体（12px），引用用户原消息的前 30 字符。这样回放时能清楚看到"这段回复是因为用户问了进度"。

**时间轴 minimap 上的标记**：排队消息在 minimap 上用一个小三角标记（`slate-400`，向下箭头），hover 显示消息预览。和模型降级/错误的三角标记形状相同但颜色不同，视觉语言一致。

## Memory 页面

### 设计思路

Memory 和 Session 的使用方式完全不同。Session 是线性的时间线回放，Memory 是**非线性的知识网络**。核心挑战：记忆数量会越来越多，类型多样（session/incident/runbook/decision/note），而且彼此之间有关联。需要**搜索能力 + 结构化浏览 + 关联可导航**。

### 页面结构

左右分栏（master-detail 同屏）：左侧列表 40%，右侧详情 60%。

和 Session 页面不同，Memory 用左右分栏而不是列表→详情的页面跳转——浏览记忆时经常需要快速扫多条，左右分栏减少页面跳转的打断感。右侧占比更大是因为记忆内容本身（Markdown 文档）需要更多阅读空间。

```
┌─ Memory ───────────────────────────────────────────────────────┐
│  ┌─ 搜索 + 过滤 ────────────────────────────────────────────┐  │
│  │  🔍 搜索记忆...                                    ⌘ K   │  │
│  │  类型: 全部 · incident · runbook · decision · note        │  │
│  │  状态: 全部 · verified · draft · conflict                 │  │
│  │  排序: 相关度 · 最近更新 · 最近创建 · confidence ↓        │  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌─ 记忆列表 ──────────────────────┬─ 记忆详情 ──────────────┐ │
│  │                                  │                         │ │
│  └──────────────────────────────────┴─────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

### 搜索交互

- **语义搜索**（默认）：自然语言输入，走 Embedding 向量检索
- **精确搜索**：引号包裹时全文匹配，如 `"EADDRINUSE"`
- 输入即搜索，300ms 防抖
- 全局快捷键 `⌘ K`（在任何页面按下跳转到 Memory 搜索）

### 过滤器

类型颜色（全站统一）：

### 记忆详情面板

四个区块：

**头部** — 类型、标题、元数据。元数据用一个紧凑的 2×2 网格展示（状态/confidence/创建时间/更新时间/来源），背景用比面板更深一层的色块，视觉上像一个"信息卡"嵌在面板里。

**内容** — Markdown 渲染。这是面板的主体，占最多空间。代码块、列表、标题按标准 Markdown 样式渲染。

**关联** — 这条记忆的上下文网络：

- **来源 Session**：点击跳转到 Session 回放页面
- **关联记忆**：`related` 字段指向的其他记忆，点击直接在左侧列表中定位并加载

**操作栏** — 底部固定，三个操作按钮：

| 操作 | 说明 |

### 设计决策

**全页编辑器，无分栏**。Memo 就是一个文件，不需要列表→详情的导航。CodeMirror + Markdown 高亮，打开即编辑，`⌘ S` 保存（也支持自动保存，5 秒防抖）。

**编辑器样式**和 Memory 编辑模式的 CodeMirror 保持一致——同样的字体（Geist Mono 13px）、同样的 Markdown 语法高亮配色。

**Agent 分区的视觉区分**。虽然底层是纯 Markdown，编辑器里可以在 `### Agent` 标题处加一条极淡的左侧色条（`cyan-400` 10% 不透明度），视觉上暗示"这是 Agent 的区域"。人类可以编辑任何位置，色条只是辅助认知，不是限制。

**实时协作提示**。如果 AI 正在编辑这个文件（通过文件锁检测），底部状态栏显示：

```

│  ⟳ Coder Agent 正在编辑...    最后保存: 3 分钟前              │
└──────────────────────────────────────────────────────────────┘
```

不做实时协同编辑（不是 Google Docs），但要让用户知道"AI 也在改这个文件"。如果检测到冲突（用户保存时文件已被 AI 修改），弹出简单的 diff 确认。

**底部状态栏**：40px，固定底部。显示最后编辑者（人类还是哪个 Agent）、时间、保存状态。信息密度低，不需要抢注意力。

```
│  │  ─────────────────────────────────────────────────────── │   │
│  │  10:05:23    info  sess_001    tool_call    bash ls -la  │   │
│  │  10:05:20    info  sess_001    tool_call    edit src/..  │   │
│  │  10:04:58    warn  system      api_timeout  opus 超时    │   │
│  │  10:04:55    info  sess_001    llm_request  claude-son.. │   │
│  │  10:04:30    info  sess_002    tool_call    fetch ..     │   │
│  │  ...                                                     │   │
│  │                                                          │   │
│  └──────────────────────────────────────────────────────────┘   │

│  │    "output_summary": "listed 12 files",                  │   │
│  │    "duration_ms": 45                                     │   │
│  │  }                                                       │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 设计决策

**表格为主**。日志是结构化数据，表格是最高效的呈现方式。每行单行显示，Geist Mono 13px，行高紧凑（32px）。表格全宽，不分栏——需要最大水平空间来展示信息。

**三种日志来源切换**。对应 `.zero/logs/` 下的三个 JSONL 文件：

| 来源 | 说明 | 显示列 |
|------|------|--------|
| `operations` | 工具调用记录（默认，最常用） | tool / input / output |
| `requests` | LLM 请求记录 | model / tokens / cost |
| `snapshots` | 上下文快照 | trigger / tools / messages_before |

切换时表格列动态变化。

**级别用颜色标记**。`error` 行整行微弱红色背景（`red-400` 5% 不透明度），`warn` 用 `amber-400` 5%，`info` 无背景。在密集的日志表格里，颜色是最快的扫描锚点。

**时间范围**。默认"最近 1 小时"，可选"最近 24 小时"、"最近 7 天"、"自定义范围"。日志量可能很大，不预加载全量。

**行展开详情**。点击某行，下方原位展开一个面板（不是跳转、不是侧边栏），显示该条日志的完整 JSON。JSON 语法高亮，代码块样式和全站一致。`Esc` 或再次点击收起。

**实时尾部追踪**。右上角 `▶ Live` 开关。打开时新日志实时从底部推入（WebSocket），自动滚动。行为像 `tail -f`——开发者最熟悉的模式。用户上翻时自动暂停追踪，出现"跳到最新 ↓"。

**搜索过滤**。`⌘ F` 聚焦到过滤输入框，支持对任意列做文本过滤。输入即过滤，200ms 防抖。

#### Provider 操作

**添加 Provider**：点击底部「+ 添加 Provider」，展开内联表单。表单根据选择的认证方式动态切换凭证输入区：

```
┌─ 添加 Provider ──────────────────────────────────────────────┐
│  名称         [                    ]                          │
│  协议类型     [Anthropic Messages ▾]                          │
│               可选: Anthropic Messages / OpenAI Chat          │
│               Completions / OpenAI Responses                  │
│  Base URL     [https://api.anthropic.com    ]                 │
│                                                               │
│  认证方式     ○ API Key  ○ OAuth Token  ○ ChatGPT OAuth       │
│                                                               │
│  ┄┄┄┄┄┄┄┄ 凭证区（根据认证方式切换）┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄   │
│                                                               │
│  [API Key 模式]                                               │
│  API Key      [                    ]    存入保密箱             │
│                                                               │
│  [OAuth Token 模式]                                           │
│  OAuth Token  [                    ]    存入保密箱             │
│  ℹ 粘贴 Anthropic OAuth Token（sk-ant-oat- 开头）             │
│                                                               │
│  [ChatGPT OAuth 模式]                                         │
│  [🔗 通过 ChatGPT 登录授权]                                   │
│  ℹ 点击后打开浏览器窗口完成 OAuth 登录                        │
│                                                               │
│                                           取消  ·  添加       │
└───────────────────────────────────────────────────────────────┘
```

- 协议类型下拉选择三种之一（详见 [Architecture - Provider Adapter]）
- 选择协议类型后自动填充默认 Base URL，用户可覆盖
- **认证方式**用 Radio 切换（详见 [Architecture - 认证方式]）：
  - `API Key`：粘贴输入，存入保密箱。Anthropic 协议使用 `x-api-key`，OpenAI 兼容协议使用 `Authorization: Bearer`
  - `OAuth Token`：粘贴输入（`sk-ant-oat-` 前缀），存入保密箱。系统通过 Anthropic SDK 的 `authToken` 参数注入认证并附加 Beta Headers（详见 [TechStack - Anthropic OAuth Token]）。适用于 Anthropic OAuth Token
  - `ChatGPT OAuth`：点击按钮打开浏览器 OAuth 授权页面。用户完成登录后，系统通过回调自动获取 Token 并存入保密箱。按钮在授权完成前显示「🔗 通过 ChatGPT 登录授权」，完成后切换为「✓ 已授权」（`emerald-400`）+ 「重新授权」链接
- API Key / OAuth Token 输入后显示为 `●●●●●●` 遮蔽，旁边显示凭证引用名
- 表单校验：名称不可重复，Base URL 格式校验，ChatGPT OAuth 模式下必须完成授权后才能点击「添加」

**编辑 Provider**：点击 Provider 卡片右上角「编辑」，卡片头部区域原位切换为可编辑表单（和 Memory 编辑模式的过渡方式一致——输入框从文字位置展开，不重绘整个卡片）。可修改 Base URL 和凭证。协议类型、认证方式和名称创建后不可修改（变更这些等价于删除重建）。

**删除 Provider**：点击「删除」，弹出确认对话框，显示该 Provider 下所有模型列表以及受影响的降级链。如果降级链中包含该 Provider 的模型，需先从降级链中移除。确认后删除 Provider 及其下所有模型。

#### 模型操作

**添加模型**：点击 Provider 卡片内「+ 添加模型」，在模型列表底部展开内联表单：

```
┌─ 添加模型 ────────────────────────────────────────────────────┐
│  别名           [                    ]    如 claude-opus       │
│  Model ID       [                    ]    如 claude-opus-4-6   │
│  Max Context    [200000    ]                                   │
│  Max Output     [32000     ]                                   │
│  Capabilities   [tools] [vision] [reasoning] [+ 添加]          │
│  Tags           [powerful] [+ 添加]                            │
│  Pricing        □ 自定义价格（不勾选则使用 litellm 默认值）     │
│                                                                │
│                                            取消  ·  添加       │
└────────────────────────────────────────────────────────────────┘
```

- Capabilities 和 Tags 用胶囊标签，`×` 删除，`+` 添加
- Pricing 默认折叠（大多数模型不需要自定义），勾选后展开 input/output/cache_write/cache_read/reasoning 五个价格字段
- 别名不可与同 Provider 下其他模型重复

**编辑模型**：点击模型行右侧「编辑」，该行展开为可编辑表单（和添加表单结构相同，预填当前值）。别名创建后不可修改。

**删除模型**：模型行 hover 时右侧出现 `×`，点击需确认。如果模型在降级链中或是当前活跃模型，提示用户先调整。

#### 凭证状态指示

每个 Provider 卡片头部显示认证方式和凭证状态。格式为"认证: {方式} · {状态}"：

| 认证方式 | 状态 | 显示 | 说明 |
|---------|------|------|------|
| 所有 | `● 已配置` | `emerald-400` | 保密箱中有对应凭证 |
| 所有 | `○ 未配置` | `red-400` | 缺少凭证，该 Provider 下所有模型不可用 |
| 所有 | `● 已配置 · 不可达` | `amber-400` | 有凭证但健康检查失败（网络问题或 Base URL 错误） |
| OAuth | `● 已配置 · 已过期` | `amber-400` | Token 过期（健康检查 401/403），需重新授权 |
| ChatGPT OAuth | `⟳ 授权中...` | `cyan-400` | OAuth 浏览器窗口已打开，等待用户完成授权 |

点击状态标签的行为根据认证方式不同：

- **API Key / OAuth Token**：弹出凭证编辑弹窗（预填遮蔽值），可更新或清除
- **ChatGPT OAuth**：弹出弹窗显示"重新授权"按钮，点击后重新发起 OAuth 流程

当系统通过 WebSocket 收到 `credential:expired` 事件时，对应 Provider 卡片的凭证状态自动切换为"已过期"，并在卡片顶部显示一条内联警告："凭证已过期，请重新配置"（`amber-400` 背景横幅）。

#### 模型健康检查

Models Tab 加载时和每次保存后，自动对所有配置了凭证的 Provider 执行一次轻量健康检查（调用 Provider Adapter 的 `healthCheck()`，详见 [TechStack - Provider Adapter 接口]）。健康检查同时验证网络可达性和凭证有效性。结果显示在 Provider 卡片上：

- 检查中：Provider 名称旁边显示 `⟳` 旋转图标
- 可达且凭证有效：不显示额外标记（干净状态）
- 不可达：Provider 名称旁边显示 `⚠ 不可达`（`amber-400`），hover 显示错误详情
- 凭证失效（401/403）：凭证状态切换为「已过期」，提示用户重新配置

**高级选项**：默认折叠——大多数定时任务用默认值就够了。点击展开显示任务重叠策略、错过执行策略、超时设置。

### 设计决策

**堆叠条形图**。用 recharts，Y 轴费用（$），X 轴日期。每个柱子按模型分色堆叠。模型颜色取品牌色的不同明度变体——都在 cyan 色域内，不引入新色调：

| 模型 | 颜色 |
|------|------|
| claude-opus | `cyan-400` (`#22d3ee`) |
| claude-sonnet | `cyan-600` (`#0891b2`) |
| gpt-4o | `cyan-800` (`#155e75`) |
| 其他 | `slate-500` |

同色系渐变，视觉统一，不会变成彩虹。

**Token 用量图**和费用趋势同结构，Y 轴单位变成 k tokens。两张图并排，用户可以对照"钱花在哪了"和"Token 消耗在哪了"。

**模型分布**用水平条形图，和 Dashboard 费用概览的条形图样式一致。

**详细记录表格**。底部按日期 × 模型的维度聚合，展示输入/输出/缓存/费用的明细。从 `metrics.db` 查出。Geist Mono，紧凑表格。

**时间范围**。7d / 30d / 90d 快捷选择 + 自定义。切换时图表平滑过渡（数据点 crossfade，不要整个图重绘闪烁）。

## Chat Drawer（Web Channel）

从右侧滑出的轻量对话面板。Web UI 的主要角色是观测和管理，对话是辅助功能，用抽屉保持上下文不丢失。

### 触发和布局

```
┌─ 主界面 ─────────────────────────────────────────┬─ Chat ────┐
│                                                    │           │
│  (Dashboard / Sessions / Memory / ...)             │  ┌─────┐ │
│                                                    │  │ 💬  │ │
│  ← 主内容区自动收窄                                │  │     │ │
│                                                    │  │ ... │ │
│                                                    │  │     │ │
│                                                    │  │ 输入 │ │
│                                                    │  └─────┘ │
└────────────────────────────────────────────────────┴──────────┘
                                                     360px
```

**触发方式**：侧边栏底部聊天图标按钮。点击后 Drawer 从右侧滑入（`cubic-bezier(0.16, 1, 0.3, 1)`，300ms），主内容区同步收窄。再次点击或 `Esc` 收起。

**宽度**：360px 固定。足够容纳一行中文或代码，但不过宽影响主内容。

**不遮挡**：Drawer 是推挤式（不是浮层）——主内容区收窄而不是被盖住。用户可以一边看 Dashboard 一边对话。

### Drawer 内部结构

```
┌─ Chat Drawer (360px) ────────────────────┐
│  ┌─ 头部 ──────────────────────────────┐ │
│  │  Web Channel  ·  claude-opus    ×   │ │
│  └─────────────────────────────────────┘ │
│  ┌─ 消息区 ────────────────────────────┐ │
│  │                                      │ │
│  │  系统通知（如果有）                   │ │
│  │  ┌────────────────────────────────┐  │ │
│  │  │ 📋 daily_update_check 完成     │  │ │
│  │  │ 未发现新版本                    │  │ │
│  │  └────────────────────────────────┘  │ │
│  │                                      │ │
│  │  用户: 刚才那个更新检查正常吗？       │ │
│  │                                      │ │
│  │  AI: 是的，今天凌晨 2 点的定时...     │ │
│  │                                      │ │
│  └─────────────────────────────────────┘ │
│  ┌─ 输入区 ────────────────────────────┐ │
│  │  [输入消息...              ] ↵ 发送  │ │
│  └─────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

**头部**：显示"Web Channel" + 当前模型名，`×` 关闭。极简。

**消息区**：对话消息和系统通知共存（和架构里的通知机制一致）。通知用卡片样式（浅色边框 + 图标），对话用和 Session 回放一致的气泡区分——用户消息有 `cyan-400` 左边框，AI 消息无边框。

**输入区**：单行输入框，`Enter` 发送，`Shift+Enter` 换行。高度自适应（最多 4 行后出现滚动条）。右侧发送按钮。

**流式响应**：AI 回复实时流入，和其他 IM Channel 行为一致。

**工具调用**：在消息区显示折叠的工具调用块，样式和 Session 回放一致——但因为空间小，默认只显示工具名和一行摘要。

**Session 管理**：Drawer 对应一个 `source: web` 的 Session。关闭 Drawer 不结束 Session，重新打开时恢复上下文。`/new` 命令创建新 Session。

### 排队消息反馈

所有 Channel（飞书、Telegram、Web Chat Drawer）对排队消息使用统一的反馈模式：

**即时确认**：用户发送消息后，如果 Agent 正忙（tool use loop 中），Channel 立即返回一条轻量确认。确认方式因 Channel 而异：

| Channel | 确认方式 | 说明 |
|---------|---------|------|
| 飞书 | 消息回复 "⏳ 已收到，任务处理中，稍后回复" | 用飞书的消息回复功能，关联到用户原消息 |
| Telegram | 消息回复 "⏳ 已收到，任务处理中，稍后回复" | 用 Telegram 的 reply_to_message |
| Web Chat Drawer | 消息气泡下方灰色提示 | 和 Session 详情页一致 |

确认消息用统一文案，保持简短。确认消息**不进入 Session 的 messages 数组**（它是 Channel 层的即时反馈，不是 Agent 的回复）。

**处理后通知**：Agent 回应排队消息后，回复通过正常的流式推送路径送达 Channel。在 IM 端，回复同样以 reply 形式关联到用户的排队消息，让对话线索清晰。

### 费用计算

模型价格不额外存储，从 [litellm/model_prices_and_context_window.json](https://github.com/BerriAI/litellm/blob/main/model_prices_and_context_window.json) 默认值或 `config.yaml` 覆盖值获取，单位为 `$/M tokens`。每条请求的实际费用在写入 `requests.jsonl` 时已根据 token 用量和模型价格计算并记录（详见 [Architecture - 模型定价与用量追踪]）。

```typescript
// TokenUsage 和 ModelPricing 类型定义详见 [TechStack - Provider Adapter 接口]
function calcCost(tokens: number, pricePerM: number): number {
  return (tokens / 1_000_000) * pricePerM
}

function calcRecordCost(usage: TokenUsage, pricing: ModelPricing): number {
  const perM = (tokens: number, price?: number) =>
    price ? (tokens / 1_000_000) * price : 0
  return (
    perM(usage.input, pricing.input) +
    perM(usage.output, pricing.output) +
    perM(usage.cacheWrite ?? 0, pricing.cacheWrite) +
    perM(usage.cacheRead ?? 0, pricing.cacheRead) +
    perM(usage.reasoning ?? 0, pricing.reasoning)
  )
}
```

## 设计原则总结

1. **安静时简洁，有事时醒目** — Dashboard 动态布局，正常时不显示待办卡片
2. **信息密度适中** — 不是极端高密度，有呼吸感的间距
3. **材质感** — 磨砂玻璃、极细边框、微光 hover，不是平面色块
4. **有意义的运动** — 每个动画都传达信息（渐次浮现引导视线、脉搏表示活跃、数字滚动表示消耗）
5. **类型系统** — 颜色绑定类型（incident=红、runbook=绿），全站统一
6. **关联可导航** — Memory 的关联记忆跳转、Session 的来源链接，形成知识网络
7. **实时感知** — WebSocket 推送，Dashboard 事件流、活跃 Session 工具调用、实时 Session 回放都是动态的
8. **键盘友好** — Session 回放、Memory 浏览都支持键盘导航
