# ZeRo OS Web UI 设计规范

## 定位

Web UI 是 ZeRo OS 的控制面板 + 观测台，同时作为轻量 Channel。

- **主要功能**：观测系统状态、查看历史、管理配置
- **次要功能**：Web Channel（侧边抽屉形式的轻量对话）
- 飞书 / Telegram 是主力交互入口，Web UI 是辅助

---

## 视觉风格：Calm Futurism

安静的未来感——不是赛博朋克的霓虹喧嚣，而是深空望远镜控制台的气质。Linear 的克制 + Vercel 的质感 + 天文台控制室的氛围。

### 色彩系统

**背景层级（冷蓝灰调）**

| 层级 | 色值 | 用途 |
|------|------|------|
| 深层背景 | `#09090b` | zinc-950 领域，接近纯黑但偏冷 |
| 主背景 | `#0f1117` | 深邃墨蓝灰 |
| 卡片表面 | `#161b22` | 浮起感 |
| 浮层/弹出 | `#1c2333` | 最亮背景层 |

卡片用磨砂玻璃效果，加 inner border 和 inner shadow 模拟物理边缘折射：

```css
background: rgba(22, 27, 34, 0.7);
backdrop-filter: blur(16px) saturate(1.2);
border: 1px solid rgba(255, 255, 255, 0.06);
box-shadow: inset 0 1px 0 rgba(255, 255, 255, 0.06);
```

**强调色（品牌色：冷青色）**

全站只用一个强调色调（cyan），通过明度变化区分层级。不引入第二色调。

| 用途 | 色值 | 说明 |
|------|------|------|
| 主强调 | `#22d3ee` | cyan-400，系统"脉搏色" |
| 强调悬浮 | `#06b6d4` | cyan-500 |
| 强调暗淡 | `#0e7490` | cyan-700 |
| 强调光晕 | `rgba(34, 211, 238, 0.08)` | 背景辉光 |

**状态色**

| 错误/熔断 | `#f87171` | red-400 |

**文字层级**

### 字体策略：双字体系统

```
显示/正文:  Geist (Vercel 出品)
            权重 400 (正文) / 600 (小标题) / 700 (大标题)

数据字体:   Geist Mono
            用于: Token 数字、Session ID、时间戳、日志、命令、YAML

中文回退:   "PingFang SC", "Noto Sans SC"
```

Serif 字体在本项目中全面禁用——Dashboard / Software UI 语境下只使用 Sans-Serif。

**字号体系**

| 用途 | 字号 | 字体/权重 | 备注 |
|------|------|-----------|------|
| 页面标题 | 20px | Geist 700 | `tracking-tight` |
| 区块标题 | 14px | Geist 600 | |
| 正文 | 13px | Geist 400 | |

**卡片 hover 效果**

```css
.card {
  transition: border-color 0.15s cubic-bezier(0.16, 1, 0.3, 1),

  box-shadow:
    inset 0 1px 0 rgba(255, 255, 255, 0.06),
    0 0 20px rgba(34, 211, 238, 0.04);

```

**卡片 active 状态**

```css
.card:active {
  transform: scale(0.98);
}
```

**页面背景纹理**

```css
body {
  background-color: #09090b;
  background-image:
    radial-gradient(circle at 20% 50%, rgba(34, 211, 238, 0.03) 0%, transparent 50%),
    radial-gradient(circle at 80% 20%, rgba(34, 211, 238, 0.02) 0%, transparent 40%);

```

**输入框**

```css
.input {
  background: rgba(15, 17, 23, 0.8);
```

### 空间布局

| 元素 | 尺寸 |
|------|------|
| 侧边栏 | 220px 固定 |
| 主内容区 | 剩余空间，`max-w-[1400px] mx-auto` |
| Chat Drawer | 360px 宽，右侧滑出 |
| 卡片间距 | 16px（`gap-4`） |
| 卡片圆角 | 12px（`rounded-xl`） |
| 内边距 | 16-20px（`p-4` / `p-5`） |

布局一律使用 CSS Grid，不使用 flexbox 百分比计算。典型的 Dashboard 双列用 `grid grid-cols-1 lg:grid-cols-2 gap-4`。

### 响应式策略

| 断点 | 宽度 | 布局变化 |
|------|------|----------|
| `< 768px` | 手机/小平板 | 侧边栏折叠为底部 Tab Bar，Chat Drawer 全屏覆盖，所有多列布局回落单列 |
| `768px - 1024px` | 平板 | 侧边栏收窄为 icon-only（56px），Memory 分栏变为上下堆叠 |
| `> 1024px` | 桌面 | 完整布局 |

移动端全高区域使用 `min-h-[100dvh]`，禁止 `h-screen`。

### 动画原则：有意义的运动

全站动画统一使用 `cubic-bezier(0.16, 1, 0.3, 1)`（Apple 弹性缓出）。仅动画 `transform` 和 `opacity`，禁止动画 `top` / `left` / `width` / `height`。

**页面进入 — 渐次浮现（Staggered Reveal）**

```css
.card {
  animation: fadeUp 0.4s cubic-bezier(0.16, 1, 0.3, 1) forwards;
}
.card:nth-child(n) { animation-delay: calc(n * 60ms); }

@keyframes fadeUp {
  from { opacity: 0; transform: translateY(8px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

**实时数据 — Token 数字滚动**

数字更新时每位独立滚动，像机械翻牌器，让"系统正在消耗资源"可感知。

**活跃 Session — 呼吸脉搏**

```css
.status-active {
  animation: pulse 2.5s ease-in-out infinite;
}
@keyframes pulse {
  0%, 100% { box-shadow: 0 0 6px rgba(52, 211, 153, 0.4); }
  50%      { box-shadow: 0 0 12px rgba(52, 211, 153, 0.6); }
}
```

**Chat Drawer — 侧滑 + 磨砂遮罩**

```css
.drawer-panel {
  animation: slideIn 0.3s cubic-bezier(0.16, 1, 0.3, 1) forwards;
}
```

**不做动画的地方**

- 页面切换（直接渲染，不要过渡延迟）
- 表格排序/筛选（数据变更要即时）
- Hover 状态（`0.15s cubic-bezier(0.16, 1, 0.3, 1)` 足够）

### 交互状态完备性

每个可交互组件必须实现完整的状态循环：

| 状态 | 实现 |
|------|------|
| **Loading** | Skeleton loader 匹配真实布局尺寸，微光扫过动画（`shimmer`）。禁止通用圆形 spinner |
| **Empty** | 构图精致的空状态插画 + 引导文案，说明如何填充数据 |
| **Error** | 行内错误提示，`red-400` 文字 + 微弱红色背景，附带重试操作 |
| **Active/Hover** | `border-color` 或 `background` 过渡，`0.15s` |
| **Pressed** | `scale(0.98)` 或 `translateY(1px)` 模拟物理按压 |

## Dashboard 页面

### 信息层级（从上到下，从紧急到背景）

1. 系统还活着吗？（0.5 秒确认）
2. 有没有需要处理的？（1 秒扫到）
3. 现在在干什么？（活跃 Session）
4. 刚才发生了什么？（事件流）
5. 花了多少钱？（费用概览）

### A. 系统状态栏

页面顶部窄带（40px），正常时几乎隐形，异常时醒目：

```
正常:    ● Running    claude-opus    Uptime 3d 4h    Heartbeat 3s ago    3 sessions active
降级:    ◐ Degraded   claude-sonnet (降级自 opus)                        ← amber-400
修复中:  ◐ Repairing  自动修复中 (2/5)                                   ← amber-400，脉搏
熔断:    ◉ Fused      等待人工介入 · 查看 incident →                     ← red-400，脉搏加快
```

- `Heartbeat` 指标实时展示最近一次心跳距今时间（架构：主程序每 10 秒写一次心跳）。正常时 `slate-500` 文字几乎隐形，超过 30 秒变 `amber-400`，超过 50 秒变 `red-400`。
- 修复中状态（`Repairing`）展示当前修复尝试次数（如 `2/5`），点击展开当前修复步骤摘要。
- 熔断状态（`Fused`）提供 `查看 incident` 链接，跳转到 Memory 中对应的 incident 记忆。

### B. 需要关注

从备忘录"需要用户处理"和系统通知提取的待办项。**没有待办时不显示**，有待办时浮现。

五类通知场景（架构定义）全部覆盖：

```
┌─ 需要关注 ─────────────────────────────────────────────────────┐
│  [Warning]  需要授权：飞书 Bot 申请访问通讯录权限                  │
│             来自 Ops Agent · 刚刚                     [去处理]   │
│                                                                  │
│  [Warning]  需要扫码：微信公众号登录已过期                         │
│             来自 Writer Agent · 5 分钟前              [去处理]   │
│                                                                  │
│  [Warning]  Telegram Bot Token 还未提供                           │
│             来自备忘录 · 2 天前                                    │
│                                                                  │
│  [Warning]  claude-opus API 连续 3 次超时，已降级到 sonnet         │
│             自动降级 · 10 分钟前                                   │
│                                                                  │
│  [Warning]  自动修复连续 3 次失败，已触发熔断                      │
│             系统已回滚到 v1.1 stable · 1 小时前       [查看详情]  │
└────────────────────────────────────────────────────────────────────┘
```

- 卡片边框用 `amber-400` 微弱光晕，第一眼就能注意到。
- "需要扫码"和"需要授权"这类需要用户即时操作的通知，`[去处理]` 按钮用 `cyan-400` 实心，视觉权重更高。同时触发浏览器原生 Notification（需用户授权）。
- 自我修复失败的通知，`[查看详情]` 跳转到对应 incident 记忆。

### D. 活跃 Sessions

```
┌─ Active Sessions ──────────────────────────────────────────────┐
│  ┌───────────────────────────────────────────────────────────┐ │
│  │ ●  sess_001                                               │ │
│  │    飞书 · claude-opus · 开始于 10:00                       │ │
│  │    重构 Provider Adapter                                   │ │
│  │    ▸ bash  git diff --stat                    just now    │ │
│  │    ▸ edit  src/provider.ts                    2 min ago   │ │
│  └───────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

- 活跃 Session（`●`）有脉搏动画，空闲（`◌`）无
- 工具调用实时更新，新调用从底部淡入
- 工具图标 `▸` 用 `slate-500`，工具名用 `cyan-400`
- Browser 被某个 Session 占用时，其他 Session 的 Browser 调用显示 `[Lock] 等待锁释放` 标记（`slate-500`）
- 点击跳转到 Session 详情页

### E. 事件流

```
┌─ Recent Activity ──────────────────────────────────────────────┐
│  10:05   sess_001  ▸ bash     git commit -m "refactor adapter" │
│  10:04   sess_001  ▸ edit     src/provider.ts L42-58           │
│  09:30   system    ▸ repair   自动修复成功：重启主进程           │
│  09:28   system    ▸ repair   检测到心跳超时，开始诊断           │
│  09:00   sess_003  ▸ sched    weekly_report 触发               │
│  08:30   system    ▸ alert    claude-opus API 超时，降级到 sonnet│
│  02:05   system    ▸ version  自动 commit: refactor adapter    │
│                                          查看全部 →             │
└────────────────────────────────────────────────────────────────┘
```

- 单行时间线，信息密度高
- 新事件从顶部滑入（`transform: translateY(-12px)` → `translateY(0)`）
- 系统级事件用状态色高亮
- 新增事件类型：`repair`（自我修复步骤）、`version`（Git commit）、`fuse`（熔断触发/恢复）
- 实时更新（WebSocket）

### F. Channel 状态

当任何 Channel 断连时，在"需要关注"和费用概览之间插入一行紧凑的 Channel 状态条：

```
┌─ Channels ─────────────────────────────────────────────────────┐
│  [Plugs] 飞书 ● 在线    Telegram ● 在线    Web ● 在线          │
└────────────────────────────────────────────────────────────────┘
```

全部在线时用 `slate-500` 文字，几乎隐形。任何 Channel 离线时，该 Channel 名变 `red-400`，整条边框变 `red-400` 微弱光晕。全部在线时可折叠隐藏（保持 Dashboard 简洁），有异常时自动浮现。

### 动态布局

Dashboard 根据系统状态自适应：

- 一切正常时：B 和 F 不出现，页面简洁
- 有待办时：B 和 C 并排（`grid grid-cols-1 lg:grid-cols-2 gap-4`）
- Channel 异常时：F 插入在 B 和 C 之间
- 系统异常（修复中/熔断）时：B 提升到全宽，状态栏变色

### Dashboard 空状态

系统刚安装、没有任何 Session 记录时：

```
┌────────────────────────────────────────────────────────────────┐
│                                                                 │
│   ZeRo OS 已就绪                                                │
│                                                                 │
│   通过飞书、Telegram 或右下角对话框                               │
│   发送第一条消息开始使用                                          │
│                                                                 │
│   或在 Config 中配置定时任务                                      │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

居中排版，`slate-400` 文字，无干扰。

## Sessions 页面

### 列表视图

```
┌─ Sessions ─────────────────────────────────────────────────────┐
│  ┌─ 筛选栏 ─────────────────────────────────────────────────┐  │
│  │  状态: 全部 · 活跃 · 已完成 · 已归档                      │  │
│  │  来源: 全部 · 飞书 · Telegram · Scheduler · Web           │  │
│  │                                          [MagnifyingGlass] │  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ● sess_001         重构 Provider Adapter                │   │
│  │  飞书 · claude-opus → claude-sonnet · 今天 10:00 - 10:45 │   │
│  │  12 次工具调用 · 3.2k tokens · $0.24                     │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  ● sess_002         部署 Telegram Bot                    │   │
```

每条 Session 三行：

1. 状态灯 + ID + 任务描述（视觉焦点）

3. 工具调用次数 · Token · 费用（判断"规模"）

筛选栏新增**状态过滤**（活跃 / 已完成 / 已归档）。已归档 Session 默认不在列表中显示，需切换到"已归档"才可见。

## Memory 页面

### 设计思路

Memory 和 Session 的使用方式完全不同。Session 是线性的时间线回放，Memory 是**非线性的知识网络**。核心挑战：记忆数量会越来越多，类型多样（session/incident/runbook/decision/note），而且彼此之间有关联。需要**搜索能力 + 结构化浏览 + 关联可导航**。

### 页面结构

左右分栏（master-detail 同屏）：左侧列表 40%，右侧详情 60%。使用 `grid grid-cols-[2fr_3fr] gap-4`。

和 Session 页面不同，Memory 用左右分栏而不是列表→详情的页面跳转——浏览记忆时经常需要快速扫多条，左右分栏减少页面跳转的打断感。右侧占比更大是因为记忆内容本身（Markdown 文档）需要更多阅读空间。

移动端（`< 1024px`）：分栏变为上下堆叠，列表在上、详情在下，或者列表全屏 + 点击后推入详情页。

```
┌─ Memory ───────────────────────────────────────────────────────┐
│  ┌─ 搜索 + 过滤 ────────────────────────────────────────────┐  │
│  │  [MagnifyingGlass] 搜索记忆...                      ⌘ K   │  │
│  │  类型: 全部·incident·runbook·decision·note·inbox·preference│  │
│  │  状态: 全部 · verified · draft · conflict                 │  │
│  │  排序: 相关度 · 最近更新 · 最近创建 · confidence ↓        │  │
│  └───────────────────────────────────────────────────────────┘  │
│  ┌─ 记忆列表 ──────────────────────┬─ 记忆详情 ──────────────┐ │
│  │                                  │                         │ │
│  └──────────────────────────────────┴─────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

### 搜索交互

- **语义搜索**（默认）：自然语言输入，走 Embedding 向量检索
- **精确搜索**：引号包裹时全文匹配，如 `"EADDRINUSE"`
- 输入即搜索，300ms 防抖
- 全局快捷键 `⌘ K`（在任何页面按下跳转到 Memory 搜索）

### 过滤器

类型颜色（全站统一）：

|------|----------|
| incident | `red-400`/10% 背景 + `red-400` 文字 |
| runbook | `emerald-400`/10% + `emerald-400` |
| decision | `amber-400`/10% + `amber-400` |
| note | `blue-400`/10% + `blue-400` |
| session | `slate-400`/10% + `slate-300` |
| inbox | `cyan-400`/10% + `cyan-400` |
| preference | `violet-400`/10% + `violet-400` |

新增 `inbox`（待整理原始记录）和 `preference`（身份记忆）类型。

过滤器默认折叠成一行紧凑显示，只有在需要精细筛选时才展开完整选项。大多数时候搜索框就够用了。

### 记忆列表

每条记忆卡片三行：

```
第一行:  [类型标签]  [标题]
第二行:  [confidence 指示器]  [状态标签]  [时间]
第三行:  [tags]
```

**Confidence 指示器**：5 个圆点，填充数量代表 confidence 区间：

| 显示 | 区间 | 颜色 | 含义 |
|------|------|------|------|
| `●●●●●` | 0.8 - 1.0 | `emerald-400` | 可用于自动修复 |
| `●●●●○` | 0.6 - 0.8 | `amber-400` | |
| `●●●○○` | 0.4 - 0.6 | | |
| `●●○○○` | 0.2 - 0.4 | | |
| `●○○○○` | 0.0 - 0.2 | `red-400` | 低可信度 |

**状态标签**：

| 状态 | 样式 |
|------|------|
| `verified` | `emerald-400` 文字 |
| `draft` | `slate-400` 文字 |
| `conflict` | `red-400` 文字 + 微弱红色背景（需要关注） |
| `archived` | `slate-600` 文字（极弱化） |

**Tags**：`slate-500` 文字，`#` 前缀，不加背景框——保持轻量，tags 是辅助信息不需要视觉重量。

#### 列表交互

- 选中某条记忆时，左边框亮起 `cyan-400`，右侧面板加载详情
- 键盘导航：`j/k` 上下移动，`Enter` 选中
- 滚动加载，不做分页——记忆条目通常不会太多（几百条），虚拟列表足够

### 记忆详情面板

四个区块：

**头部** — 类型、标题、元数据。元数据用一个紧凑的 2x2 网格展示（状态/confidence/创建时间/更新时间/来源 Session），背景用比面板更深一层的色块，视觉上像一个"信息卡"嵌在面板里。

**内容** — Markdown 渲染。这是面板的主体，占最多空间。代码块、列表、标题按标准 Markdown 样式渲染。

**关联** — 这条记忆的上下文网络：

- **来源 Session**：点击跳转到 Session 回放页面（通过 `session_id` 字段关联）
- **关联记忆**：`related` 字段指向的其他记忆，点击直接在左侧列表中定位并加载
- **Tags**：点击某个 tag 等于在搜索框里输入该 tag 做过滤

关联记忆的跳转是这个页面最重要的交互之一——它让"平铺列表"变成了"可导航的知识网络"。点击一条关联记忆 → 左侧列表高亮该条 → 右侧加载新内容 → 新内容里又有新的关联 → 继续探索。

**操作栏** — 底部固定，三个操作按钮：

| 操作 | 说明 |
|------|------|
| 归档 | 将这条记忆移入 `archive/`，从主列表中隐藏 |
| 删除 | 彻底删除（需二次确认） |
| 在文件中打开 | 打开对应的 `.md` 文件路径 |

按钮用 ghost 样式（无背景，hover 时出现淡色背景），删除按钮 hover 时变红。

### 编辑模式

点击头部的"编辑"按钮后，详情面板切换为编辑模式：

- **标题**：变为可编辑输入框
- **状态**：下拉选择（verified / draft / conflict）
- **Confidence**：滑块，0-1，步长 0.05，右侧显示精确数值
- **内容**：CodeMirror Markdown 编辑器，支持语法高亮。不做分屏预览——编辑区本身就有 Markdown 高亮，够了
- **Tags**：胶囊式标签编辑，`×` 删除，`+` 添加

编辑模式和阅读模式之间的切换用平滑过渡——输入框从文字位置"展开"出来，不是整个面板重绘。

## Memo 页面

备忘录是 AI 和人类之间的**活文档**。打开这个页面的场景：想看 AI 现在在做什么/打算做什么，想调整优先级给 AI 指方向，想看有没有需要自己处理的事项。

```
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 设计决策

**全页编辑器，无分栏**。Memo 就是一个文件，不需要列表→详情的导航。CodeMirror + Markdown 高亮，打开即编辑，`⌘ S` 保存（也支持自动保存，5 秒防抖）。

**编辑器样式**和 Memory 编辑模式的 CodeMirror 保持一致——同样的字体（Geist Mono 13px）、同样的 Markdown 语法高亮配色。

**Agent 分区的视觉区分**。虽然底层是纯 Markdown，编辑器里可以在 `### Agent` 标题处加一条极淡的左侧色条（`cyan-400` 10% 不透明度），视觉上暗示"这是 Agent 的区域"。人类可以编辑任何位置，色条只是辅助认知，不是限制。

**实时协作提示**。如果 AI 正在编辑这个文件（通过文件锁检测），底部状态栏显示：

```
┌──────────────────────────────────────────────────────────────┐
│  ⟳ Coder Agent 正在编辑...    最后保存: 3 分钟前              │
└──────────────────────────────────────────────────────────────┘
```

不做实时协同编辑（不是 Google Docs），但要让用户知道"AI 也在改这个文件"。如果检测到冲突（用户保存时文件已被 AI 修改），弹出简单的 diff 确认。

**底部状态栏**：40px，固定底部。显示最后编辑者（人类还是哪个 Agent）、时间、保存状态。信息密度低，不需要抢注意力。

### 设计决策

**表格为主**。日志是结构化数据，表格是最高效的呈现方式。每行单行显示，Geist Mono 13px，行高紧凑（32px）。表格全宽，不分栏——需要最大水平空间来展示信息。

**四种日志来源切换**。新增 `trace`：

| 来源 | 说明 | 显示列 |
|------|------|--------|
| `operations` | 工具调用记录（默认，最常用） | tool / input / output |
| `requests` | LLM 请求记录 | model / tokens / cost |
| `snapshots` | 上下文快照 | trigger / tools / messages_before |
| `trace` | 调用链追踪 | session / 瀑布图缩略 / 总耗时 |

`trace` 来源以 Session 为粒度，每行是一个 Session 的完整调用链。点击展开为瀑布图（横轴时间，每步一条柱子，颜色按工具类型），用于事后分析复杂任务的耗时分布。

**级别用颜色标记**。`error` 行整行微弱红色背景（`red-400` 5% 不透明度），`warn` 用 `amber-400` 5%，`info` 无背景。在密集的日志表格里，颜色是最快的扫描锚点。

**时间范围**。默认"最近 1 小时"，可选"最近 24 小时"、"最近 7 天"、"自定义范围"。日志量可能很大，不预加载全量。

**行展开详情**。点击某行，下方原位展开一个面板（不是跳转、不是侧边栏），显示该条日志的完整 JSON。JSON 语法高亮，代码块样式和全站一致。`Esc` 或再次点击收起。

**实时尾部追踪**。右上角 `▶ Live` 开关。打开时新日志实时从底部推入（WebSocket），自动滚动。行为像 `tail -f`——开发者最熟悉的模式。用户上翻时自动暂停追踪，出现"跳到最新"按钮。

**搜索过滤**。`⌘ F` 聚焦到过滤输入框，支持对任意列做文本过滤。输入即过滤，200ms 防抖。

## Chat Drawer（Web Channel）

从右侧滑出的轻量对话面板。Web UI 的主要角色是观测和管理，对话是辅助功能，用抽屉保持上下文不丢失。

### 触发和布局

```
┌─ 主界面 ─────────────────────────────────────────┬─ Chat ────┐
│                                                    │           │
│  (Dashboard / Sessions / Memory / ...)             │  头部     │
│                                                    │           │
│  ← 主内容区自动收窄                                │  消息区   │
│                                                    │           │
│                                                    │           │
│                                                    │  输入区   │
└────────────────────────────────────────────────────┴──────────┘
                                                     360px
```

**触发方式**：侧边栏底部 `ChatCircle` 图标按钮。点击后 Drawer 从右侧滑入（`cubic-bezier(0.16, 1, 0.3, 1)`，300ms），主内容区同步收窄。再次点击或 `Esc` 收起。

**宽度**：360px 固定。足够容纳一行中文或代码，但不过宽影响主内容。

**不遮挡**：Drawer 是推挤式（不是浮层）——主内容区收窄而不是被盖住。用户可以一边看 Dashboard 一边对话。

**移动端**（`< 768px`）：Drawer 全屏覆盖，从底部滑入。

### Drawer 内部结构

```
┌─ Chat Drawer (360px) ────────────────────┐
│  ┌─ 头部 ──────────────────────────────┐ │
│  │  Web Channel  ·  claude-opus  [X]   │ │
│  └─────────────────────────────────────┘ │
│  ┌─ 消息区 ────────────────────────────┐ │
│  │                                      │ │
│  │  系统通知（如果有）                   │ │
│  │  ┌────────────────────────────────┐  │ │
│  │  │ [ClipboardText]                │  │ │
│  │  │ daily_update_check 完成        │  │ │
│  │  │ 未发现新版本                    │  │ │
│  │  └────────────────────────────────┘  │ │
│  │                                      │ │
│  │  用户: 刚才那个更新检查正常吗？       │ │
│  │                                      │ │
│  │  AI: 是的，今天凌晨 2 点的定时...     │ │
│  │                                      │ │
│  └─────────────────────────────────────┘ │
│  ┌─ 输入区 ────────────────────────────┐ │
│  │  [输入消息...     ] [PaperPlaneRight]│ │
│  └─────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

**头部**：显示"Web Channel" + 当前模型名，`[X]` 关闭。极简。

**消息区**：对话消息和系统通知共存（和架构里的通知机制一致）。通知用卡片样式（浅色边框 + Phosphor icon），对话用和 Session 回放一致的气泡区分——用户消息有 `cyan-400` 左边框，AI 消息无边框。

**输入区**：单行输入框，`Enter` 发送，`Shift+Enter` 换行。高度自适应（最多 4 行后出现滚动条）。右侧 `PaperPlaneRight` 发送按钮。

**流式响应**：AI 回复实时流入，和其他 IM Channel 行为一致。

**工具调用**：在消息区显示折叠的工具调用块，样式和 Session 回放一致——但因为空间小，默认只显示工具名和一行摘要。

**Session 管理**：Drawer 对应一个 `source: web` 的 Session。关闭 Drawer 不结束 Session，重新打开时恢复上下文。支持两个命令：

| 命令 | 说明 |
|------|------|
| `/new [model]` | 创建新 Session，可选指定模型 |
| `/model [model]` | 携带参数则切换模型；不携带参数则返回当前模型名称 |

### 费用计算

费用不存储，查询时实时算。价格以 `$/M tokens` 存于 `config.yaml`，默认取自 [litellm/model_prices_and_context_window.json](https://github.com/BerriAI/litellm/blob/main/model_prices_and_context_window.json)。

```typescript
function calcCost(tokens: number, pricePerM: number): number {
  return (tokens / 1_000_000) * pricePerM
}

function calcRecordCost(usage: TokenUsage, pricing: ModelPricing): number {
  const perM = (tokens: number, price?: number) =>
    price ? (tokens / 1_000_000) * price : 0
  return (
    perM(usage.input_tokens, pricing.input) +
    perM(usage.output_tokens, pricing.output) +
    perM(usage.cache_write_tokens, pricing.cache_write) +
    perM(usage.cache_read_tokens, pricing.cache_read) +
    perM(usage.reasoning_tokens, pricing.reasoning)
  )
}
```

## 设计原则总结

1. **安静时简洁，有事时醒目** — Dashboard 动态布局，正常时不显示待办和 Channel 卡片
2. **信息密度适中** — 不是极端高密度，有呼吸感的间距（`VISUAL_DENSITY: 4`）
3. **材质感** — 磨砂玻璃 + inner border + inner shadow、极细边框、微光 hover，不是平面色块
4. **有意义的运动** — 每个动画都传达信息（渐次浮现引导视线、脉搏表示活跃、数字滚动表示消耗），统一 `cubic-bezier(0.16, 1, 0.3, 1)` 缓出
5. **类型系统** — 颜色绑定类型（incident=红、runbook=绿），全站统一
6. **关联可导航** — Memory 的关联记忆跳转、Session 的来源链接，形成知识网络
7. **实时感知** — WebSocket 推送，Dashboard 事件流、活跃 Session 工具调用、心跳、Channel 状态都是动态的
8. **键盘友好** — Session 回放、Memory 浏览都支持键盘导航
9. **状态完备** — 每个页面定义 Loading（skeleton）、Empty（引导）、Error（内联重试）三态
10. **无 emoji** — 所有图标使用 Phosphor Icons，统一 `strokeWidth: 1.5`
11. **响应式** — 三档断点（手机/平板/桌面），移动端单列 + 底部 Tab Bar
12. **架构对齐** — UI 覆盖架构文档中的所有用户可见模块：自我修复状态、版本管理、工具管理、保密箱、Channel 管理、任务编排、上下文压缩、Trace
